AlgorithmSolution_CraneDrawDoll

Algorithm - 크레인 인형 뽑기

문제 설명

  • 게임개발자인 죠르디는 크레인 인형뽑기 기계를 모바일 게임으로 만들려고 합니다.
    죠르디는 게임의 재미를 높이기 위해 화면 구성과 규칙을 다음과 같이 게임 로직에 반영하려고 합니다.

  • 게임 화면은 1 x 1 크기의 칸들로 이루어진 N x N 크기의 정사각 격자이며 위쪽에는 크레인이 있고 오른쪽에는 바구니가 있습니다. (위 그림은 5 x 5 크기의 예시입니다). 각 격자 칸에는 다양한 인형이 들어 있으며 인형이 없는 칸은 빈칸입니다. 모든 인형은 1 x 1 크기의 격자 한 칸을 차지하며 격자의 가장 아래 칸부터 차곡차곡 쌓여 있습니다. 게임 사용자는 크레인을 좌우로 움직여서 멈춘 위치에서 가장 위에 있는 인형을 집어 올릴 수 있습니다. 집어 올린 인형은 바구니에 쌓이게 되는 데, 이때 바구니의 가장 아래 칸부터 인형이 순서대로 쌓이게 됩니다. 다음 그림은 [1번, 5번, 3번] 위치에서 순서대로 인형을 집어 올려 바구니에 담은 모습입니다.

  • 만약 같은 모양의 인형 두 개가 바구니에 연속해서 쌓이게 되면 두 인형은 터뜨려지면서 바구니에서 사라지게 됩니다. 위 상태에서 이어서 [5번] 위치에서 인형을 집어 바구니에 쌓으면 같은 모양 인형 두 개가 없어집니다.

  • 크레인 작동 시 인형이 집어지지 않는 경우는 없으나 만약 인형이 없는 곳에서 크레인을 작동시키는 경우에는 아무런 일도 일어나지 않습니다. 또한 바구니는 모든 인형이 들어갈 수 있을 만큼 충분히 크다고 가정합니다. (그림에서는 화면표시 제약으로 5칸만으로 표현하였음)

  • 게임 화면의 격자의 상태가 담긴 2차원 배열 board와 인형을 집기 위해 크레인을 작동시킨 위치가 담긴 배열 moves가 매개변수로 주어질 때, 크레인을 모두 작동시킨 후 터트려져 사라진 인형의 개수를 return 하도록 solution 함수를 완성해주세요.

제한사항

  • board 배열은 2차원 배열로 크기는 5 x 5 이상 30 x 30 이하입니다.
  • board의 각 칸에는 0 이상 100 이하인 정수가 담겨있습니다.
  • 0은 빈 칸을 나타냅니다.
  • 1 ~ 100의 각 숫자는 각기 다른 인형의 모양을 의미하며 같은 숫자는 같은 모양의 인형을 나타냅니다.
  • moves 배열의 크기는 1 이상 1,000 이하입니다.
  • moves 배열 각 원소들의 값은 1 이상이며 board 배열의 가로 크기 이하인 자연수입니다.

입출력 예

board moves result
[[0,0,0,0,0],[0,0,1,0,3],[0,2,5,0,1],[4,2,4,4,2],[3,5,1,3,1]] [1,5,3,5,1,2,1,4] 4

입출력 예에 대한 설명

입출력 예 #1

  • 인형의 처음 상태는 문제에 주어진 예시와 같습니다. 크레인이 [1, 5, 3, 5, 1, 2, 1, 4] 번 위치에서 차례대로 인형을 집어서 바구니에 옮겨 담은 후, 상태는 아래 그림과 같으며 바구니에 담는 과정에서 터트려져 사라진 인형은 4개 입니다.

구현 코드

내가 구현 한 방법

  • 결과 값은 똑같이 나오지만 프로그래머스에서 채점을 했을 때 오답으로 나온다.

코드 설명

  • 각 작은 배열들이 들어가는 Arraylist를 만들어서 값을 넣어주고 ArrayList를 모아두는 큰 ArrayList를 또 만들어서
  • 그 안에서 반복하면서 moves[i]와 큰 Arraylist의 인덱스와 moves[i] 같다면
  • 작은 Arraylist의 요소를 제거해서 또 새로운 Arraylist를 만들어서 거기에 값을 넣고
  • 마지막에 새로운 Arraylist를 반복하면서 같은 값들이 연속해서 나오면 answer을 증가시킨다

다른 사람의 구현 방법

  • 내가 한 것 보다 훨씬 간결하고, 테스트가 잘 통과된다.

  • 먼저 Stack을 만들어 주고, 처음에 값 0을 넣어준다.

  • foreach 문으로 moves 배열을 반복하면서 값들이 나온다. 그리고 안에 반복문을 board의 길이만큼 돌면서 크레인이 좌우로 왔다 갔다하면서 인형을 뽑아 내는거니깐

  • 열을 move-1로 해준다 move를 -1해주는 이유는 배열의 인덱스는 04인데 들어오는 값의 범위는 15까지여서 -1씩해서 배열의 인덱스와 맞춰주기 위해서이다.

  • 이렇게 board[i][move-1]이라고 해주고 이동한 열에서 인형을 뽑을 때 i(행)를 반복해서 한칸씩 밑으로 내려가면서 값이 0이 아닐 때,

  • stack.peek()와 board[i][move-1]이 같다면, 똑같은 값이 연속으로 만나는거니깐 폭팔해서 사라져야해서 stack의 peek 값을 빼주고
    사라진 인형의 수를 세야하니깐 answer를 2씩 증가시켜준다.

  • stack.peek()와 board[i][move-1]이 똑같지 않다면, 새로운 인형이므로 stack에 값을 넣어준다.

  • 그러고 나서 board에서 가져왔던 board[i][move-1]의 값을 0으로 초기화 시켜주고 반복문을 빠져나오게 했다.

    • 그 값은 이미 뽑힌 인형이라고 생각하고 0일 때는 빈칸을 의미하기 때문에 뽑힌 값을 0으로 처리해주는 것이다.

Java_Handling_Exception

예외 처리(Handling Exceptions)

오류와 예외

  • 프로그램이 동작 중에 비정상 종료가 발생하는 상황
  • 심각도에 따라 오류와 예외로 분류할 수 있음

오류(Errors)

  • 시스템 수준에서 발생하여, 개발자가 조치할 수 없는 수준을 의미한다.
  • 시스템의 메모리가 부족하거나, 프로그램이 꼬이는 경우, 무한히 메소드 호출이 발생하는 등 프로그램 복구가 불가능한 상황.
    • 메모리 부족 현상 - 힙영역 new로 객체를 많이 만들거나, stack을 너무 많이 사용하게 되면 메소드 콜이 너무 많이 누적될 경우
            (자기 자신이 무한하게 호출될 수 있어서 ,반복이 많이 발생할 수 있어서 재귀함수를 잘 못 사용하면 메모리 부족이 발생한다.)
    • 스택 영역은 일정 크기만큼 제한이 있고, 힙은 물리적으로 메모리가 크다. 그래서 메모리 부족은 스택 영역에서 주로 발생한다.
    • 프로그램이 꼬이는 경우 - 메모리 호출이 끝나고 앞에 과정을 찾을 수 없을 수 있다.
      스택이 터질 경우에 스택 메모리가 손상이 되면서 메소드가 끝나면서 어디 메소드로 돌아가야 하는지 찾기 어려울 수 있다.
  • 프로그램을 진행하기 어려운 상태, 계속했다가 해킹을 할 수 있는 여지가 생긴다. 오류가 발생하면 프로그램을 종료하는게 정상적인 방법이다.
    소스코드를 보고 버그픽스를 해서 문제를 해결하는 것이 해결 방법이다.

예외(Exceptions)

  • 개발자가 구현한 로직에서 발생한 것으로 개발자가 다른 방식으로 처리가 가능한 것들이다.
  • 오류에 비해 심각도가 낮으며, 프로그램 정상적인 흐름만 방해하는 경우이다.
    • 파일명으로 파일을 읽으려 하는데, 파일을 찾을 수 없을 때,
    • 네트워크 연결이 유실 (끊겼을 때) 다시 연결 해야할 때 등이다.
  • 문제 상황을 해결하는 로직을 별도로 구성하여 런타임에서 자연스럽게 해결 가능하다.

예외 처리

  • 예외가 발생했을 때, 이 상황을 감지하고 해결하기 위한 동작하는 코드
  • try ~ catch 구문과 Exception 클래스와 그 자식 클래스를 활용.

예외처리를 하는 이유

  • 안드로이드나 ios App에 아이디 비밀번호를 써서 로그인할 경우를 가정해서,
    비밀번호를 입력할 때 비밀번호가 틀려서 튕기는 현상이 발생한다. 또 비밀번호를 써서 로그인 하려고 하는데,
    또 틀려서 또 튕긴다. 그러다가 로그인을 했는데 컨텐츠서버가 반응이 늦어서 터지게 되서
    다시 로그인해야하는 이런 상황이 발생할 수 있는데 그래서 예외처리를 해야한다.

  • 백엔드에선 서버 Downtime(반응을 할 수 없는 시간을 말함)이 중요하다. 서비스를 서버에 올리면 서비스는 계속 돌아가야 한다.
    왜냐하면 그래야 외부에서 요청이 들어왔을 때 바로 대답을 해줄 수 있으니깐!
    서버 재부팅을 할 때 서버 다운타임이 발생한다. 이런 다운타임을 최소화 하는게 중요하다 이런걸 예외처리를 잘하는게 중요한 이유이다.

Throwable 클래스

  • Throwable 클래스는 예외 처리의 최상위 클래스이다. Exception 클래스와 Error 클래스는 Throwable 클래스를 상속 받는다.
    • Exception
      • Checked Exceptions : 예외 처리되지 않으면 컴파일이 되지 않는 예외
        • 예외 처리가 반드시 되어야 하는 자주 발생하는 예외가 Checked Exception으로 된다.
      • Unchecked Exceptions : 예외가 처리되지 않아도 컴파일이 되는 예외
        • 특별히 문제가 될 것 같다하면 예외를 따로 처리를 해주면 되는 것이다
          처리 될 수도 있고 안될 수도 있는데 처리 되면 안전한 프로그램이다.
    • Error : 프로그램이 복구 불가능한 상황

Exception 클래스

  • Throwable 클래스의 주요 메소드

    메소드 설명
    public String getMessage() 발생한 예외에 대한 메세지를 반환
    public String getLocalizedMessage() 오버라이드하여 추가 메세지 제공(오버라이드하지 않으면 getMessage()와 동일
    public Throwable getCause() 예외의 원인이 되는 Throwable 객체 반환
    public void printStackTrace() 예외가 발생된 메소드가 호출될 때의 Method call stack을 출력

  • printStackTrace()

    • 프로그램을 실행 하고 예외가 발생 했을 때 무슨 메소드의 몇 번째 줄에 어떤 예외가 발생 했다 라고
      메세지가 나오고 어떤 메소드 였다면 메소드 내부의 내용을 넣고 어떤 문제가 있다고
      메소드 콜 스택을 전부다 보여 준다. 스택을 추적을 하는 형식으로 흔적을 다 보여준다.

예외 처리 기법

try ~ catch 구문

  • try 블럭 : 예외가 발생할 수 있는 코드들을 작성하는 영역이다.

  • catch 블럭 : 예외가 발생할 경우 예외가 어떤 클래스(예외는 클래스이다!)들의 자손인지 확인하고 받을 수 있는지
    확인하여 받을 수 있으면 받아서 예외처리를 하는 영역이다.

    다중 예외 처리

    • 여러 개의 catch 구문을 사용하면 다중 예외를 처리할 수 있음.
    • if ~ else if 구문처럼, 순차적으로 검사하면서 적용 가능한 예외를 처리
    • 다형성이 적용되어, 자식 예외를 처리 가능
    • Exception 클래스는 모든 예외의 상위 클래스이므로 하위의 예외 클래스들을 먼저 써줘서
      예외처리 부분을 만들고 마지막에 Exception 클래스를 사용해서 나머지 예외를 처리하도록 구현해야 한다.

    try ~ catch ~finally 구문

    • try 구문 실행 중에 어떤 일이 발생해도 반드시 실행되어야 하는 구문은 finally 블록에 작성
    • try 구문 내에 return문이 있는 경우에도 finally 블록은 실행됨
    • try 구문 내에서 접근한 System 자원을 안전하게 복구하기 위해 사용

finally를 지원하는 이유

  • try블럭에서 리소스를 할당한 경우라면, fileInputStream은 fileName 파일을 오픈해서 파일을 읽어서 정보를 가져오는 스트림을 생성하는 것이다.
    스트림은 닫아 주기 전까지 메모리에 남아있다.
    new로 만든 객체가 생성이 되어서 힙영역에 메모리가 형성되고, gc로 회수되기 전까지 객체는 살아남아있다.

  • 이 경우 두가지 문제가 있는데

    • 첫째, fileName이라는 파일을 오픈해서 실제로 메모리를 많이 잡아먹을 수 있다.
    • 둘째, 이 파일의 리소스를 점유해버리는 문제가 생길 수 있다.
      • 점유해버린다는 건 프로그램을 계속 실행중인 상태를 말하는 것이다.
      • 윈도우에서 프로그램을 실행 중일 때 그 파일을 변경하거나 삭제할 수 없는 현상이 발생하는 걸 본 적이 있을 텐데 여기서 그런 문제가 발생할 수 있다.

  • try 내에서 파일을 열고 닫고를 다 해버린다는 코드를 작성해 놨지만, try 중간에 에러가 발생하면 닫지 않고 catch로 그냥 넘어갈 수 있다
    그래서 finally에서 catch로 넘어갔든 안갔든 리소스를 사용하는 객체가 발생하면 그 객체를 검사해서 객체가 리소스를 가지고 있다면 해제를 해준다.

  • 이러한 문제 때문에 fianlly를 지원한다.

try ~ with ~ resources 구문

  • 위의 stream을 열고 닫고 해야하는데 read할 때도 예외처리를 해줬는데 또 close도 예외처리를 해줘야하는 번거로움이 발생한다.
    이러한 성가신 코드를 쓰지 않기 위해서 추가된 기능

  • Java 1.7에서 추가된 기능

  • AutoClosable 인터페이스를 구현하는 리소스를 자동으로 close 처리

    예외 처리 위임

    throws

  • 호출한 메소드로 예외 처리를 전달하는 방식. 호출한 메소드는 예외처리를 해줘야 한다.

  • Checked Exception의 경우 throws로 위임 가능하나, 최종적으로 try ~ catch를 만나야 함

  • Unchecked Exception의 경우 throws로 위임하지 않아도 자동으로 전달

  • 메소드를 오버라이드한 경우, 조상 클래스의 메소드보다 조상 예외는 던질 수 없음

    • BarThree 클래스는 Foo 클래스를 상속 받고 있는데, BarThree methodA()메소드의 예외는 Foo클래스 methoadA의 예외보다 더 상위예외를 던질 수 없다는 의미이다.
    • 자식 클래스의 메소드 예외 < 부모 클래스의 메소드 예외

      throw

  • 예외를 발생시키는 키워드

  • new 키워드로 새 Exception 객체를 생성하여 예외 내용을 작성

  • 어떤 특정 메소드가 Checked Exception을 발생시킨다면 throws해서 (throws뒤에) Checked Exception을 적어줘야 한다.
    어떤 상황이 발생했을 때, Exception을 던질 거면 throw를 이용해서 객체를 만들어서 던져줘야한다.
    Exception은 객체이고 결국 객체를 만들어야한다. new를 이용해서 객체를 만들고 throw 해주면 이 코드가 작성된 메소드에 throws를 했기 때문에
    메소드를 콜하는 바로 상위 메소드든 그 상위 메소드든 언젠가 메인까지 올라가다보면 Checked Exception이라면
    try~catch를 만나게 된다. 거기에 catch에 throw해준 예외의 부모 클래스가 있다면 catch를 해서 예외처리를 해줄 것이다.
    getMessage를 담아서 throw를 해주는 것이다.

throw를 사용하는 상황?

  • throw 는 내가 하고싶을 때 하면된다. 어떤 에러 사항이 있는데 프로그램이 진행이 되다가 throw를 한다는건,
    이미 구현된 메소드 내부적으로 (빌트인된 쪽에서, 메소드 안에서) Exception이 발생하지 않았는데
    내가 Exception을 발생시키고 싶으면 throw를 쓸 수 있는 것이다. 내가 판단했을 때 어떤 일이 생기면 Exception이 발생해야하는 상황이다라고 정해줄 수 있다.
    이럴 때 사용한다.

*프로그램 작성하는데 비밀번호를 틀린걸 알게 됬는데 이걸 그냥 로직으로 처리할 수 있지만, 비밀번호가 틀린걸 예외로 볼 수 있다.
비밀번호가 틀린상황이 if문 조건에 잡혔을 때 이때 사용자 정의 예외로 해서 throw new passwordWrongException(“. . .”)를 만들어서 throw해줄 수 있다.
그러면 이 예외가 발생하면 Unchecked Exception이고 try~catch로 처리가 안됬다면 프로그램이 죽을 것이다.
이걸보고 익셉션을 처리해야 하구나 생각한다. 이때 이 메소드를 콜할 때 try~catch를 구현하는 것이다.
패스워드가 틀린경우에 어떻게 해결할 건지 다시 아이디 비밀번호를 입력하게 한다던지의 로직 카운트를 해서
5번틀리면 프로그램을 꺼지게하는 처리를 하는 방식을 catch에 작성하면 된다.

사용자 정의 예외(Custom Exceptions)

  • Exception 또는 RuntimeException 클래스를 상속하여 작성
    • Exception을 상속한 경우 Checked Exception이 되어 반드시 예외를 처리해야 한다.
  • throwable 을 상속하는 Exception이 있고, Exception을 상속하는 RuntimeException이 있다. Exception을 상속해서 만드는 예외들은 Checked Exception이다.
    그래서 Checked Exception을 만들고 싶다면 Exception을 상속하면 되고 Unchecked Exception를 하고 싶으면 RuntimeException을 상속해서 만들면 된다.
    그러면 Unchecked Exception이 됨. 재밌는건 RuntimeException도 Exception을 상속하고 있다.
    원리적으로 보면 다 Checked Exception인데 RuntimeException은 Unchecked Exception를 만드는 특수한 Exception이다.

내부 클래스(Inner Classes)

내부 클래스란

  • 연관 있는 것을 그룹 지어 사용할 때 내부 클래스, 외부 클래스가 주로 사용된다.

  • 클래스 내부에 선언하는 클래스로 중첩 클래스라고도 부른다.

  • HAS-A관계에 있는 클래스가 해당 클래스에서만 사용될 경우 주로 사용.

  • Map.Entry<String,Integer> 이게 대표적인 내부 클래스이다.

  • 내부에 클래스를 선언하는 이유는?

    • 내부에서만 사용하기 위해서 선언한다.
    • 구현할 내용이 클래스 내용를 갖고 있는데, 외부에선 쓸일이 없고 내부에서만 쓸 때 사용한다.
    • 바깥 외부에서 사용할려고 만들려고 한다면 외부에서 따로 클래스를 만드는게 맞다.
    • 그래서 내부에서만 진짜 사용한다라고 하면 내부클래스에 private 을 써주는게 맞다.

  • 내부 클래스에서 주로 사용하는 것은 익명 내부 클래스인데, 익명 내부 클래스를 사용하는 것이 람다식이여서 결국 람다식을 사용하기 위한 것

  • 클래스 안에 클래스 안에 인스턴스변수와 클래스변수가 있을 수 있고, 중첩되면서 굉장히 복잡해질 것이다.

    • 앞의 내용들의 개념을 확실히 잡은 후에 들어야 한다. 이 부분을 공부하면서 앞에 개념이 흔들린다. 특히 메모리 구조에 대한 부분이!

내부 클래스의 종류

  • 클래스 영역에 선언

    • 인스턴스 내부 클래스 : 외부 클래스의 멤버 변수처럼 사용 가능
    • 클래스 내부 클래스 : static이 붙은 정적인 내부 클래스

  • 로컬 영역에 선언

    • 로컬 내부 클래스 : 선언된 영역의 Scope 내에서 사용 가능
    • 익명 내부 클래스 : 객체를 일회성으로 생성하기 위한 이름이 없는 클래스
      • new —-() : —– 여기에 클래스 이름을 사용했어야 했는데
        클래스를 한 번만 사용할거면 이름이 없어도 된다. 일회성으로 생성을 한다면 익명으로 만드는 것이 가능하다. 이것들은 암기하는 영역이다.

다양한 내부 클래스

인스턴스 내부 클래스 (Instance Inner Class)

  • 클래스 영역에 static 키워드 없이 선언되면 인스턴스 내부 클래스이다.

  • static 클래스가 아닌 일반적인 클래스처럼 동작 한다.

  • 외부 클래스의 private을 포함한 모든 멤버에 접근 가능

  • 외부 클래스 객체를 통해서만 생성 가능

  • static 멤버 변수는 사용할 수 없으나, static final 은 사용 가능

    • static 멤버 변수가 사용이 될려면 클래스가 먼저 있고 그 안에 static 멤버 변수가 존재하게 되는데,
      문제는 인스턴스 내부 클래스는 외부 클래스의 인스턴스가 만들어져야
      그 안에 내부 클래스가 생기는 것이기 때문에 static 멤버 변수가 있을 수 없다.

    • static 변수를 각 객체가 따로 가져가야 하나, 아니면 인스턴스가 달라도 같은 내부클래스라면 같은 static 변수를 가져가야하나 애매해진다.
      같은 메모리를 공유해야 하는지, 아니면 다른 메모리를 써야하는지 애매해진다. 애매한 걸 없애서 static 변수를 사용할 수 없게 했다.

    • static final은 사용 가능 - 변하지 않는 상수이기 때문에 사용이 가능하다.
      상수는 클래스의 생성과 별개로 먼저 상수 풀에 저장이 된다. 먼저 생성이 되어 있기 때문에 어디에서 사용을 하던 사용할 수 있는것 같다.(더 자세히 이해를 해야한다.)

  • 외부 클래스의 객체를 만들고 객체 내에서만 사용이 가능하다 -> 이해 되면 이해 안되면 암기

e

  • 클래스이기 때문에 이외에는 클래스처럼 사용이 가능하다. 이너 메소드가 존재할 수 있다.
  • outer 클래스의 멤버 변수도 inner 메소드에서 사용이가능 private이여도 접근이 가능하다.
  • inner 클래스는 인스턴스에 속해 있기 때문에 속해 있는 인스턴스 변수에 접근이 가능하다.
  • inner 클래스를 outer 클래스를 통해서만 객체 생성이 가능하다는 것은
    • outerMethod는 static이 붙어있지 않기 때문에 인스턴스 멤버 메소드이다. outerMethod를 call하기 위해서 outer 인스턴스가 있다는 것 이므로
    • outer 인스턴스에 속하는 메소드이기 때문에 inner 클래스를 바로 사용이 가능하다.(생성 가능) 그런 다음 메소드 콜이 가능하다.

    outerMethod는 객체.outerMethod로 선언이 되기 때문에 이미 객체가 있다는 전제하에 outerMethod 바디에 내부클래스를 생성할 수 있음
  • 반면 메인메소드는 스태틱이므로 인스턴스가 아니라 클래스 메소드이기 때문에 outer.new InnerClass를 사용해서 생성해야 한다.

클래스 내부 클래스 (Class Inner Class)

  • 클래스 영역에 static 키워드와 함께 선언된 내부 클래스

  • 외부 클래스 객체가 없어도 생성 가능

  • 외부 클래스의 private을 포함한 모든 멤버에 접근 가능

  • static 멤버 변수를 가질 수 있음.

  • 인스턴스 내부클래스와 상당히 유사하지만 객체를 있고,없고의 차이 스태틱을 사용할 수 있고 없고의 차이.

  • 외부 클래스의 특정 인스턴스에 속한 클래스가 아니기 때문에 외부 클래스에 접근하려면 내부클래스에서 외부클래스를 생성을 해야 접근이 가능하다.

  • 외부 클래스의 메소드에서 내부 클래스의 객체를 다 만들 수 있음.

  • 외부 클래스 내부 클래스는 독립적인 클래스가 맞긴한데 종속 독립의 개념보다 내부클래스는 외부클래스의 사정을 알고있는 클래스이다?

  • 클래스 내부 클래스의 내부 클래스에서 외부 클래스의 객체를 만들면 외부 클래스의 priavte에 접근이 가능하므로 독립된 것 처럼 사용할 수 있지만 독립된건 아니다.

  • 위에 인스턴스 내부 클래스 예제에서는 자연스럽게 외부클래스의 멤버변수에 접근하는게 자연스러웠지만,

  • 클래스 내부 클래스에서는 클래스 내부 클래스가 존재하더라도 innerMethod()가 정의가 되는 시점에 아우터 클래스의 객체가 없다.

  • 그래서 아우터 클래스의 객체에 접근할려면 외부에서 넣어준던지 어그리게이션으로 넣어주던지

  • 내부클래스의 외부클래스 객체를 생성해야만 외부클래스의 private 변수에 접근이 가능한 이상한 논리가 생성이 된다.

  • 받아 들이시면 됩니다~ 이렇게 하는 경우가 별로 없어요~

  • 내부에서 외부를 참조하는건 드물다 굳이 이렇게 동작을 한다.

  • 객체가 없다라는걸 이해하는게 중요하다

  • 메인에서 Outer 클래스로 객체를 만들 때 아우터 클래스 객체 안에 Inner 클래스 객체가 있는 것이 아니다.
    컴포지션이랑은 전혀 다른 개념이다 클래스는 있는데 객체는 없는 것이다.

  • Outer 클래스로 객체를 만들 때 Inner 클래스의 메소드를 사용할 수 없다. Outer 클래스에서 Outer.ClassInner 형태로 내부클래스의 static만 접근이 가능하다.

로컬 내부 클래스(Local Inner Class)

  • 각각이 서로 다른 블럭에 영향을 주지 않는다.
  • 바깥 쪽에 있는 클래스의 멤버변수, static 변수에 접근할 수 있다.
  • 클래스 영역이 아닌 로컬 영역에 선언된 클래스이다.
    • 메소드, 생성자, 초기화 블록의 내부
  • 로컬 내부 클래스의 라이프 사이클은 해당하는 블록 내에서만 존재한다.
  • 스택 메모리에 정의가 된다.
  • 정적 변수는 가질 수 없으나, static final 변수는 가질 수 있다.
  • 로컬 영역의 변수 다음과 같은 사항을 따른다.
    • Java 1.7 : final로 선언된 변수에만 접근 가능하다.
    • Java 1.8 : 모든 변수에 접근 가능하나, final로 간주되어 새로운 값을 할당하는게 불가하다.

      로컬 내부 클래스의 복잡한(신기한?) 활용 방법

  • 제목을 어떻게 지어야 할지 모르겠는데 로컬 내부 클래스를 사용했을 때 신기한 상황이 전개 될 수 있는데 그 예를 보자!

익명의 내부 클래스 (Anonymous Inner Class)

  • 로컬 내부클래스와 똑같다 똑같은데 이름이 없는 것이다.
  • 딱 한번만 쓸 수있다.
  • 추상클래스나 인터페이스의 구현에 많이 사용. 추상클래스나 인터페이스에 대한 바로 생성을 할 수 있다.
    • 원래는 추상클래스나 인터페이스를 생성할 때 상속하는 클래스를 만들고 그 클래스를 생성해서 사용했다.
    • 단 하나의 추상클래스나 인터페이스는 클래스 이름 없이 바로 new 키워드로 생성 가능.
  • 안드로이드 프로그래밍에서 많이 사용된다.(말 들으니깐 모르고썼던 부분들이 기억이 난다.)

자료의 입출력(Inputs and Outputs)

출력 메소드

  • 포맷 문자열을 이용한 문자열/기본형 출력
  • 객체를 직접 출력 -> toString() 재정의를 통해 어떻게 출력할 것 인지를 구현할 수 있다.

printStream의 메소드

  • print():

    • public void print(Object obj)
    • obj를 스트림으로 출력한다.

  • println():

    • public void println(Object x)
    • x를 스트림으로 출력하고 줄바꿈한다.

  • printf() :

    • public PrintStream printf(String format, Object … args)

    • Format 맞추어 args를 출력한다.

    • 정수 포맷팅

    • 실수 포맷팅

      입력 메소드

  • Scanner 클래스를 이용하여 입력 받을 수 있다.

  • System.in은 키보드 입력을 받는 것이다 .PC에서 사용하기 때문에 표준입력을 이것을 사용하는 것이다.

  • Scanner 클래스의 주요 메소드

    • next(): 공백을 기준으로 한 단어씩 입력 받는다.
    • nextLine(): 한 줄 전체를 입력 받는다. (enter로 구별)
    • nextInt(): (공백을 기준으로) int 값을 입력 받는다.
    • nextDouble : double 값을 입력 받는다.
    • close() : 입력 스트림을 종료한다.
      • close() 를 사용하지 않으면 메모리 낭비가 될 수 있다,
        gc가 제대로 동작이 되지 않았을 때 사용을 신경써 주는게 좋다.
    • 값을 여러개 입력받을 때, nextInt()는 공백을 기준으로 값을 입력받기 때문에, 만약 10이라는 값을 입력하고 enter를 누르면
      10이라는 값은 출력이 되는데, enter도 하나의 값이여서 값이 남아있게 된다. 그래서 sc.nextLine()으로 enter 값을 입력받게 해게 해서
      없애준 다음에 다음 값들을 변수에 담아 주도록 해야 한다.
